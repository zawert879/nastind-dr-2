# START_HERE_WEB.md — Инициализация Svelte AI Dev System
> Помести этот файл в корень Svelte проекта.
> Прочитай его один раз при старте работы с проектом — потом работай через CLAUDE.md.

---

## Что это такое

Это система для разработки фронтенд-приложений на SvelteKit с AI-агентом (Claude).  
Вдохновлена подходами из статей о Claude Code, SKILL.md и GStack.

**Ключевая идея**: агент не держит контекст между сессиями → кодируем знания в файлы, а процессы в скилы.

**Архитектурное ограничение**: это приложение — **только фронтенд (SPA/SSG)**. Бэкенд разрабатывается отдельно. SvelteKit используется как клиентский фреймворк с роутингом. Серверные фичи SvelteKit (`+page.server.ts`, `+server.ts`, form actions, SSR) **не используются**. Взаимодействие с бэкендом — через HTTP-клиент (`fetch` / обёртки в `src/lib/api/`).

## Шаги инициализации (сделать один раз)

### 1. Заполнить project-reference.md
Открой `.claude/rules/project-reference.md` и заполни:
- Версию Svelte / SvelteKit
- CSS-подход (Tailwind / vanilla / UnoCSS)
- URL и документацию внешнего бэкенда
- Хостинг / деплой (Vercel / Cloudflare / статика)
- Выбранные паттерны (stores, клиентский роутинг)
- Способ авторизации (токены, cookies от бэкенда)

### 2. Заполнить PRD
Открой `docs/PRD.md` — заполни хотя бы:
- Цель продукта и целевая аудитория
- Ключевые фичи (скоуп MVP)
- Пользовательские сценарии
- Требования к UI/UX

Затем синхронизируй `.claude/rules/prd-summary.md` — сожми до главного.

### 3. Заполнить dev-plan.md
Открой `.claude/memory/dev-plan.md`:
- Заполни "Текущий фокус"
- Добавь задачи Фазы 0 под свой проект
- Определи первый спринт

### 4. Настроить проект
```bash
# Создать SvelteKit проект (если ещё нет)
npx sv create my-app
cd my-app
npm install

# Отключить SSR глобально — добавить в src/routes/+layout.ts:
# export const ssr = false;
# export const prerender = false;

# Проверить что dev-сервер работает
npm run dev
```

### 5. Проверить что всё работает
Напиши агенту: `/check-plan` — он должен прочитать все файлы памяти и выдать статус проекта.

## Рабочий процесс (каждый день)

```
Начало сессии:
  → Агент читает CLAUDE.md автоматически
  → /check-plan — сверить где мы

Работа над задачей:
  → /plan-feature [описание] — спланировать
  → /implement-component [задача] — реализовать
  → /review-code — проверить
  → /run-tests — прогнать тесты
  → /update-docs — синхронизировать документацию

Конец сессии:
  → Хук выведет чеклист (из settings.json)
  → Убедиться что dev-plan.md обновлён
```

## Слэш-команды (шпаргалка)

| Команда | Что делает |
|---|---|
| `/check-plan` | Срез состояния: где мы, что дальше |
| `/plan-feature [описание]` | Спланировать новую фичу до кода |
| `/implement-component [задача]` | Реализовать компонент/роут/фичу из плана |
| `/review-code` | Ревью перед коммитом |
| `/run-tests [модуль]` | Запустить тесты (Vitest / Playwright) |
| `/update-docs` | Синхронизировать всю документацию |

## Стек по умолчанию

| Слой | Технология |
|---|---|
| Фреймворк | SvelteKit (frontend-only, SSR отключён) |

## Правила эволюции системы

Эта система — живой организм. После каждого спринта спрашивай себя:

1. **Что агент делал неправильно?** → Добавь правило в CLAUDE.md или в нужный SKILL.md
2. **Что приходилось объяснять каждый раз?** → Добавь в rules/ или templates/
3. **Какой процесс был неудобным?** → Упрости или перепиши скил
4. **CLAUDE.md стал > 150 строк?** → Вынеси детали в .claude/rules/

Цель: минимум объяснений → максимум правильных действий агента.

## Философия системы

**Rules** (`.claude/rules/`) — пассивные знания. Агент читает, чтобы понять контекст.  
**Memory** (`.claude/memory/`) — живая память. Агент обновляет после каждой задачи.  
**Skills** (`.claude/skills/`) — активные процедуры. Агент запускает для выполнения типовых операций.  
**Agents** (`.claude/agents/`) — режимы мышления. Переключает контекст на нужную роль.  
**Templates** (`.claude/templates/`) — каркасы. Агент заполняет, не изобретает.
